## P0 — 11215 票据清算中账户历史不平衡约 2.6M

### 问题

截至 2026-04-11 迁移收尾,`11215 - 票据清算中 - 台州京泰` 账户净借方余额
约 **2,606,110.25 元**(1551 行 GL,Dr 64.52M / Cr 58.19M)。

按年拆分的不平衡分布(基于粗匹配,金额±日期 ±14 天):

| 年度 | BD/BP Dr | JE Cr | 年净 |
|---|---|---|---|
| 2020 | 5,157,458 | 5,242,449 | −84,991 |
| 2021 | 7,277,432 | 6,483,495 | **+793,937** |
| 2022 | 8,843,514 | 9,150,813 | −307,299 |
| 2023 | 7,422,926 | 7,304,926 | +118,000 |
| **2024** | 15,093,699 | 13,926,104 | **+1,167,595** *(已扣除本次迁移贡献)* |
| 2025 | 14,021,644 | 13,337,040 | **+684,604** |
| 2026 | 2,984,067 | 2,748,453 | +235,615 |

- 2024 / 2025 两年净 Dr 合计约 **1.85M**
- 2021 净 Dr 约 **0.79M**
- 其他年份小幅度波动

### 根因

两类情况混在一起,**暂时无法定量区分**:

1. **真的业务在途**(正常):贴现/兑付单据已在 acceptance app 做完,但银行流水
   还未在本月导入 —— 这部分下个月导入后会自然归零,不是账错
2. **历史漏记或错账**(需处理):
   - 某些 Bill Discount/Payment 产生时没有对应的银行流水 JE 跟进(录了承兑但
     没导流水,或导流水时对方科目没写 11215 清算中)
   - 某些银行流水 JE 的对方科目本该是 11215 清算中却错写到其他科目(e.g.
     其他应收款 / 其他应付款 / suspense 科目),导致 Cr 11215 缺失
   - 早期还没有"清算中"这个概念时,部分业务直接打银行叶子账户,后来用
     `migrate_bill_bank_to_clearing` 一次性挪过来,但两侧挪的力度不对称

### 已做的粗筛

- 匹配 571 对(acceptance Dr ↔ JE Cr,金额+日期一致)
- **acceptance 孤项**:233 条 / 18.85M(Dr 无对应)
- **JE 孤项**:176 条 / 12.53M(Cr 无对应)
- 双向孤项金额差 = 18.85 − 12.53 ≈ **6.32M** ≈ 净 Dr 余额

孤项明细保存在 `/tmp/orphans.json`(开发机本地,不在仓库里)。

### 需要做什么

按优先级排:

1. **财务人员配合**:从银行实际对账单出发,对照 2024 年全年的贴现和兑付流水,
   逐张核对系统里是否有对应的 Bill Discount/Payment + 对应的银行流水 JE。
   **这是复盘最权威的来源**,SQL 分析只是辅助。
2. **改进匹配算法**:现在是"金额+日期 ±14 天"的粗匹配,精确度低。下一步应
   以**票号**为主键(bill_no 在 Bill Discount/Payment 的 remarks + 大多数
   bank flow JE 的 remarks 里都有),提升匹配准确率。
3. **写一个诊断工具** `inspect_clearing_account_imbalance(year)`:
   - 按 bill_no 做精确匹配
   - 输出孤项清单(含 bill_no、金额、日期、备注)
   - 区分"应收票据一侧有但银行流水一侧无"vs"银行流水一侧有但应收票据一侧无"
   - 输出 CSV 供财务下载人工复核
4. **月末关账前跑一次**:每月关账前强制跑一次诊断 + 财务确认,把当月实际归零的
   孤项冲销,把真的在途的留到下月。

### 硬性约束

- **不要用任何自动化手段修复历史孤项**,必须财务人工确认后才能冲销,
  否则很容易把真实业务当作错账红冲掉
- **11215 清算中账户的期末余额**不是核心会计指标(它不进资产负债表),但
  长期大额 Dr 会引起审计关注,应尽快在 2026 年内归零历史包袱

## P1 — 历史迁移工具的设计缺陷(已补救但模式待内化)

### 问题

`migrate_ningbo_bank_supplier` 在处理"借壳供应商"历史数据时,用
`_delete_je_completely` 把原始 JE **整个**删掉,包括银行侧的
`Dr 10022 京泰宁波` 行。随后新建 `Bill Discount`/`Bill Payment` 时,
controller 内部又会生成一条 `Dr 10022`,看起来银行余额补回来了。
但之后跑 `normalize_bill_settlement_to_clearing` 把这条 Dr 从 10022 搬到
11215 清算中,结果 **10022 账户凭空少了 3.72M 实收资金**,与宁波银行对账
单不一致。

**已用 `create_bank_mirror_for_migrated_bills` 补救**(为 37 张单据补了
37 条 `Dr 10022 / Cr 11215` 的镜像 JE),10022 余额已回正。

### 为什么是"遗留"

补救工具是"跑完就扔"的一次性修复。**迁移工具本身的设计缺陷没有修补**。
将来再遇到类似的"借壳账户/虚拟供应商"历史数据迁移场景,仍然会踩同一个坑。

### 正确的模式

迁移工具的"删旧建新"应该是**一个原子过程**,三件事一起做:

```
对每一张旧的借壳 JE:
  1. 提取: amount, date, posted_account, counter_party
  2. 删除: 整个旧 JE (错误的 party 关系)
  3. 新建 A: Bill Discount/Payment (正规业务单据, 触发正规 GL)
  4. 新建 B: Dr 银行 / Cr 清算中 (补回银行入账记录)
  5. 提交 A 和 B, 保证 11215 清算中最终借贷抵消
```

其中 3 和 4 如果漏做任何一个,都会破坏账户余额的一致性。

### 需要做什么

1. 重构 `migrate_ningbo_bank_supplier`,把 `create_bank_mirror_for_migrated_bills`
   的逻辑内嵌进去,形成"删旧 → 建 BD/BP → 建镜像 JE"三合一流程
2. 写成可重用的基础函数 `acceptance.api.migration_utils.regulate_bypass_supplier_entries`
3. 在 `CLAUDE.md` 或 acceptance app 的 README 里加一段"借壳数据迁移的
   不变式",说明"删 JE 的同时必须补镜像 JE"

### 紧迫性

宁波银行已经处理完,短期内没有新的同类历史数据迁移需求。属于**预防性重构**,
优先级 P1 但不紧迫。下次真的遇到同类场景再改也可以。

## P1 — 缺少"银行流水导入自动按票号匹配"功能

### 问题

当前每周银行流水 xlsx 导入是**纯人工**流程(见 `docs/财务人员.md` 第 7 节):
- 财务打开 xlsx,肉眼识别哪些行是"票据贴现/兑付入账"
- 手动把对方科目从默认改成 `11215 - 票据清算中 - 台州京泰`
- 导入后手动跑 GL 报表验证清算中账户余额归零

这个流程:
- 出错率高(漏改对方科目是最常见的错误来源)
- 不可审计(人工判断没有系统留痕)
- 占用财务大量时间(每周上百条流水里可能只有几条是票据相关,但要逐行看)

### 需要做什么

实现**银行流水导入自动匹配器**:

1. **解析 remarks**:银行流水 remarks 里通常包含
   - 关键字("贴现收入""到期收款""电子商业汇票")
   - 票号(30 位数字)
   - 子票区间(带逗号的两个 12 位数字)
2. **按 bill_no 查找** acceptance 里已存在的 `Bill Discount`/`Bill Payment`,
   金额 + 日期 ±3 天匹配
3. **命中**:自动把对方科目改成 `11215 票据清算中`,标记"已匹配 → DISC-xxxxx"
4. **未命中但关键字匹配**:弹窗提醒"这笔看起来是票据相关但没找到对应
   acceptance 单据,是否要先去补录?"
5. **完全未匹配**:走原有的通用对方科目映射

这需要改银行流水导入的实现代码,目前不清楚它的具体实现位置(看起来是外部
xlsx + 标准 Data Import Tool,没有自定义 app)。需要先调研。

### 紧迫性

**中等**。实现后每月能节省财务几个小时,并根治 P0 里 2024/2025 年的孤项
累积问题的主要来源。建议下个 sprint 排进去。

## P2 — CAS 22 新金融工具准则合规性

### 问题

《企业会计准则第 22 号——金融工具确认和计量》(CAS 22,2019 年起执行)
要求按"业务模式"分类金融资产,对**银行承兑汇票**的处理与旧准则不同:

- **旧准则(2017 版)**:银承和商承都走 `应收票据`
- **新准则(CAS 22)**:
  - 持有目的"收取合同现金流量 + 出售" → `应收款项融资`(资产负债表单列)
  - 持有目的"仅收取合同现金流量" → `应收票据`(摊余成本计量)
  - 持有目的"管理流动性 + 出售" → `以公允价值计量且其变动计入当期损益的金融资产`

台州京泰作为制造业,银承主要用于**日常贴现融资**(本月承兑下月或下季贴现),
在 CAS 22 下应归入 `应收款项融资`。

当前系统:
- `acceptance/accounting_defaults.py` 硬编码 `NOTES_RECEIVABLE_ACCOUNT = "11211 - 应收票据 - 台州京泰"`
- 不管 `bill_type` 是 `Bank Acceptance Bill` 还是 `Commercial Acceptance Bill`,
  都打到同一个科目
- 商承按 CAS 22 仍走 `应收票据`,所以这部分是对的
- 银承按 CAS 22 应走 `应收款项融资`,目前不分,**有瑕疵**

### 需要做什么

1. **先确认**:台州京泰在**哪个年度**开始执行 CAS 22 新准则?这决定了是
   "从 2020 年开始追溯改"还是"从 2023 年开始改"还是"就不改"
2. 如果要改:
   - 新建科目 `11212 - 应收款项融资 - 台州京泰`
   - 在 `acceptance/accounting_defaults.py` 增加按 `bill_type` 分流的逻辑
   - 在 `BillReceive / BillTransfer / BillDiscount / BillPayment` 的 controller
     里读 `bill_of_exchange.bill_type`,动态选择 `notes_receivable_account`
   - 写一个一次性迁移工具把历史 银承 的 GL 从 `11211` 搬到 `11212`
3. 同步更新 `docs/财务人员.md` 附录 A 的科目列表

### 紧迫性

**取决于审计压力**。如果下一次外审会被卡这一条,必须做;否则可以放到 2027 年。
建议先和财务/审计沟通确认执行哪个版本准则再定。

## P2 — 科目/公司配置硬编码在 Python 常量里

### 问题

`acceptance/accounting_defaults.py` 把 `CLEARING_ACCOUNT` /
`NOTES_RECEIVABLE_ACCOUNT` / `DISCOUNT_INTEREST_ACCOUNT` / `DEFAULT_COMPANY`
全部写成硬编码的常量:

```python
CLEARING_ACCOUNT = "11215 - 票据清算中 - 台州京泰"
NOTES_RECEIVABLE_ACCOUNT = "11211 - 应收票据 - 台州京泰"
DISCOUNT_INTEREST_ACCOUNT = "财务费用_票据贴现利息 - 台州京泰"
DEFAULT_COMPANY = "台州京泰电气有限公司"
```

这些值带着公司后缀 `- 台州京泰`,导致:
- **acceptance app 与台州京泰强绑定**,不可复用到别的公司
- 科目重命名后要改代码 + 重新 build 镜像 + deploy,不能运行时配
- 测试/仿真环境要另外建同名科目才能跑起来

### 理想方案:单例 Acceptance Settings DocType

类似 `OCR Settings` 的 Single DocType,字段:
- default_clearing_account (Link to Account)
- default_notes_receivable_account_for_bank_bill (Link)
- default_notes_receivable_account_for_commercial_bill (Link)
- default_discount_interest_account (Link)
- default_company (Link)

Controller 在 validate 阶段读取这个单例,不再从 Python 常量取。

### 阻塞因素

`CLAUDE.md` 铁律 **"禁止在容器内运行任何 bench 命令,包括 bench migrate"**,
而新增 DocType 属于 schema 变更,**只能在首次安装时生效**。对于已经部署
运行的生产环境,无法通过 push + Deploy 增加新的 DocType —— 必须推倒重装
整个 site 或破例跑一次 `bench migrate`。

目前的妥协:硬编码常量是唯一无 schema 变更的方案。

### 需要做什么

等到下一次有**重大 schema 变更**(例如 CAS 22 合规改造)需要推倒重装 site
时,**顺便**把这个 Settings DocType 建出来。不值得为它单独破例 migrate。

### 紧迫性

**低**。不影响功能,只影响可维护性和可复用性。

## P2 — 贴现利息计算未用实际利率法

### 问题

`bill_discount.py::calculate_discount` 用的是最朴素的**单利/360 天**公式:

```python
self.discount_interest = self.discount_amount * (self.discount_rate / 100) * self.remaining_days / 360
```

《企业会计准则第 22 号》要求金融资产按**实际利率法**摊销贴现利息:
- 对短期票据(≤1 年),单利法的近似结果与实际利率法差异很小(分到元级别),
  实操上**可以接受**
- 对长期票据(>1 年),差异会放大,可能触达会计准则要求的精度
- 台州京泰持有的承兑绝大多数是 **6 个月期**,**不碰红线**

另外:
- **360 vs 365 vs ACT/ACT** 的惯例不统一。目前代码固定用 360,实际银行贴现
  回单有时按 365 计息。财务在 Bill Discount 页面看到的利息和银行回单可能
  差几分钱,需要**反向调利率**来让系统算的实收和回单一致
- **不支持**分阶段利率(到期前某一天以前用一个利率,之后用另一个)

### 需要做什么

优先级从低到高:

1. 把利息分母从固定的 `360` 改成 `Acceptance Settings` 里可配的 `360/365/ACT`
2. 允许用户手动覆盖 `discount_interest` / `actual_amount` 字段(目前会被
   `calculate_discount` 覆盖掉,除非开 `flags.historical_import`)
3. 实现真正的实际利率法(如果审计要求)

### 紧迫性

**低**。目前手工反推利率能对上银行回单,审计也不会卡 6 个月内的票据。
可以长期遗留。

## P3 — 票据状态机对"逆向补单"不友好

### 问题

`Bill Discount.validate_bill_status` 强制要求 BoE 状态是 `Received - Circulating`:

```python
if boe.bill_status != "Received - Circulating":
    frappe.throw(...)
```

这意味着:
- 银行流水导入发现一笔贴现款入账 → 财务查系统发现对应 BoE 状态还是
  `Received - Circulating` → 可以正常补录 ✓
- 但如果 BoE 状态已经被期初脚本直接 SQL 写成 `Discounted` 了(历史期初
  导入做过这种事) → 补 Bill Discount 被拒

今天从"宁波银行借壳单正规化"通道绕过这个检查,靠的是 `flags.historical_import`
开关。这个开关是一次性救急设计,不是长期机制。

### 需要做什么

引入**"逆向补单模式"**:在 Bill Discount/Payment 页面加一个 checkbox
"期初补录"(fieldname `is_backdated` 或 `reverse_fill`),勾选时:
- 跳过状态检查(允许 `Discounted` / `Settled` 状态下补)
- on_submit 不改 BoE 状态(因为已经是终态了,不需要再推进)
- 生成的 GL 正常

这个改动需要加 JSON schema 字段,同样被 **CLAUDE.md 的 migrate 禁令**
阻塞,同 P2 硬编码问题一样等下次重大改造时一起做。

### 紧迫性

**低**。`flags.historical_import` 作为开发专用兜底已经够用,业务场景里
一般不会出现"状态已 Settled 还要补贴现"的奇葩情况。

## P3 — 期初数据中存在同票号多条 BoE 的非标准结构

### 问题

期初 BoE 导入存在**同 bill_no 但 sub_ticket_start 重叠**的现象。例:

```
bill_no=531365101001520241129002142745
  BOE-01618: sub_ticket_start=44744585, sub_ticket_end=45414584, amount=6700
  BOE-01619: sub_ticket_start=44744585, sub_ticket_end=46244584, amount=15000
  BOE-01620: sub_ticket_start=44744585, sub_ticket_end=45851584, amount=11070
```

三条 BoE 都声称从同一个子票序号 `44744585` 起步,这在几何上是**不可能**的
(拆分后的三张子票应该占据不重叠的三段区间,例如 44744585-45414584,
45414585-46914584,46914585-48021584)。

面值之和 6700 + 15000 + 11070 = **32770** 正确对上原父票的总额,但
`sub_ticket_start` / `sub_ticket_end` 是错的 —— 说明期初导入脚本简化
处理了,没有重新分配起止号。

### 影响

- **功能层面无影响**:面值、状态、背书链条都是正常的,今天的迁移工具
  `_allocate_proportional` 已经按面值比例分摊处理
- **审计/可追溯层面有影响**:如果以后要做"按子票号追溯到具体张数"的
  报表或审计(极少见),这批数据会对不上
- **Endorsement Record 报表**里显示的子票区间可能重复展示

### 需要做什么

1. 找期初导入脚本的作者确认:这是有意简化还是 bug?
2. 如果是 bug,写一个**幂等修复脚本**重新分配 sub_ticket_start/end,
   让区间不重叠
3. 或者标记这批 BoE 为"老票/非标",在 UI 里提示用户别做细粒度拆分

### 紧迫性

**低**。影响面极窄,业务上不卡手。

## P3 — 历史 `宁波银行` 以外"借壳供应商"可能仍然存在

### 问题

老 ERP 无承兑支持时,用"虚拟供应商"绕路是一个常见套路。今天只处理了
`宁波银行` + `SUP-2070`,但**不能证明**没有其他类似的借壳。

可能的其他借壳账户名称(猜测):
- 其他银行名直接注册成 Supplier("工商银行""农业银行""中国银行")
- 抽象名称("承兑池""贴现行""票据中介")
- 内部员工/关联方名字

### 需要做什么

1. 扫描全量 Supplier 列表,用关键字 + 统计模式筛查可疑 party:
   - `supplier_name LIKE '%银行%'`
   - `supplier_name LIKE '%承兑%' OR LIKE '%贴现%' OR LIKE '%票据%'`
   - GL Entry 里 `account='22021 应付账款-结算'` 且金额大、频次高的 party
2. 人工复核可疑清单
3. 对确认是借壳的,复用 `migrate_ningbo_bank_supplier` 模板走正规化

### 紧迫性

**低,但应一次查清**。财务在下次深度盘账时(例如年末)把这件事列入清单。

## P3 — 10 笔"金额修正"FIX JE 已挪入清算中但语义未定性

### 问题

宁波银行借壳数据里有 10 笔 `remarks` 含"金额修正: 收支/金额错误"的 JE,
`cheque_no` 格式是 `F(B)-2503-003206-FIX2` 这类(原始流水 + FIX2 后缀)。
经检查原始流水 `F(B)-2503-003206` 在数据库里**不存在** —— 只有孤零零的
FIX2 条目。这说明期初导入时有意做了账务调整,但原始对应条目没进系统。

今天的 `migrate_ningbo_bank_supplier` 把这 10 笔 JE 从
`22021 应付账款-结算(宁波银行)` 搬到 `11215 票据清算中`,清掉 party,
但它们的业务含义**没有定性**:
- 既不是票据贴现也不是到期兑付
- 很可能是"某笔原始银行流水导入时记错金额/方向,用一个反冲条目修正"
- 金额总计 **1,122,545.60**(全部贷方,计入清算中 Cr)

它们目前留在清算中账户里作为"账务调整残余",没有对冲项,实际上是
**永久挂账的 Cr 1.12M**。

### 影响

这 1.12M Cr 会**长期压低** 11215 清算中账户的净 Dr 余额。从会计纯粹性
角度讲不漂亮,但不影响报表合规性(11215 本身是过渡科目,不进对外披露)。

### 需要做什么

1. 找到期初导入脚本作者或当时操作的财务,**定性**这 10 笔修正是针对什么
   错账的
2. 选项 A(最正规):找到原始错账条目,做正式的红字冲销,让两条都消失
3. 选项 B(折中):把这 10 笔从 11215 搬到一个"期初调整差异"科目
   (e.g. `以前年度损益调整` 或新建一个 `11219 期初迁移差异`)
4. 选项 C(最懒):保留现状,在财务手册里标注"11215 里有 1.12M Cr 是
   2022 年期初调整尾款,不要找对冲项"

### 紧迫性

**低**。不影响任何对外报表,不影响银行对账。可以长期留在清算中不管。

### 审计人员如果问"为什么有这么多 SQL 级别的数据修改"

准备好以下回答:

1. 历史老 ERP 不支持承兑业务,用借壳供应商绕过,2026 年上线 ERPNext 后
   需要把历史数据正规化,常规的 UI 操作无法 bulk 处理
2. 所有 SQL 级别的修改都通过 `acceptance/api/import_helpers.py` 里的白名单
   函数进行,**不直接动生产 MySQL**
3. 每一个修改工具都:
   - 支持 `dry_run` 预览
   - 支持幂等重跑
   - 在 git history 里有完整 commit 说明
   - 修改前后都有账户余额复核
4. 每次大规模数据变更都有对应的 `CLAUDE.md` 或 `docs/` 文档说明
5. 提供 `inspect_ningbo_bank_supplier` 这类**只读诊断**工具,审计人员可以
   亲自跑一遍验证状态

### 对财务人员的日常提醒

- **不要绕过** acceptance app 直接在 Journal Entry 里手工记贴现/兑付
  (会破坏状态机 + 错过 PLE 核销)
- **不要用 Administrator 直接改 GL Entry 或 Bill of Exchange 表**
  (会破坏背书链条完整性)
- 遇到不确定的情况(差一分钱、状态异常、清算中余额异常)**直接问开发**,
  不要自己 SQL 硬改
